## 一、系统概述

### 1.1 这是什么？

期货交易辩论专家团是一个**多Agent协作系统**，通过模拟真实投研团队的"数据→研究→辩论→策略→风控"流程，对商品期货品种进行系统性多空分析并输出可执行交易方案。

### 1.2 核心设计理念

```
研究员（供弹的）→ 辩手（开火的）→ 风控（叫停的）
```

**三大分工铁律**：
1. **研究员不站队** — 只列事实+边际变化，不下结论（verdict=null强制）
2. **辩手不造假** — 逻辑链必须追溯到研究员证据，禁止编数据
3. **风控不改方向** — 有否决权，无改方向权

### 1.3 与传统"多空辩论"的本质区别

| 维度 | 传统辩论（旧版） | 本系统（新版 v3.1） |
|:----|:----------------|:-------------------|
| 辩论起点 | 预设立场（多/空） | **数技师数据决定方向** |
| 正方任务 | "做多理由找一找" | "这个方向对不对？" |
| 反方任务 | "做空理由找一找" | "这个方向哪里错了？" |
| 辩论焦点 | "谁的故事讲得好" | **"这个数据信号是否可靠"** |
| 核心价值 | 各自自圆其说 | **验证数据信号的可靠性** |

### 2.3 各角色详细定义

#### 🎯 团队主管（明鉴秋）

- **职责**：选定辩论品种+周期+账户权益假设；最终拍板（执行/搁置/重辩）
- **出现时机**：仅在T-60min（选题）和T+25~T+30（拍板）出现
- **铁律**：辩论过程中不发表意见，避免带偏辩手
- **输出**：`{subject: "RB", timeframe: "daily", equity: 1000000}`

#### 📡 数技师（数技源）

- **职责**：采集K线数据、计算技术指标、输出方向判定
- **工作模式**：调用 `quant-daily/scan_all.py --symbols`（CLI或库函数）
- **不做分析**：只保证数据干净、口径一致、时效对
- **输出方向**：BUY/SELL + 等级（STRONG/WATCH/WEAK/NOISE）
- **数据溯源**：输出自带 `_meta`/`data_manifest` 字段

#### 🟢 基本面研究员（探源）

- **立场**：**严格中立**，不下多空结论
- **输出**：供给/需求/库存/利润/基差月差快照
- **约束**：
  - `verdict: null` 强制校验——含verdict即不合格
  - 每条数据标注来源+日期
  - 库存分绝对量和分位数
  - 期限结构标清contango/back
- **何时被call**：辩论中裁判或辩手要求补充特定维度数据

#### 🟢 技术面研究员（观澜）

- **立场**：**严格中立**，不下多空结论
- **输出**：趋势/关键位/量价/资金/背离/形态
- **约束**：
  - `verdict: null` 强制校验
  - 突破是否带量、持仓变化、背离级别必须标注
- **特别价值**：戳穿辩手的"假突破叙事"

#### 🔵 正方辩手（证真）

- **立场**：**信号捍卫者**——捍卫数技师给出的方向
- **核心原则**：立场不预设为多或空，而是"数技师说多我就论证多，说空我就论证空"
- **6维度分析框架**：趋势结构、量价关系、期限结构、产业链验证、基本面/市场情绪、风险点
- **场景分离**：baseline + upside 情景
- **必须包含认错信号**：什么条件下承认方向错了
- **约束**：
  - 每维度evidence带来源+日期
  - 禁止"好的方面"开头
  - 空方有效反驳必须正面接

#### 🔴 反方辩手（慎思）

- **立场**：**信号挑战者**——站在数技师方向的对立面
- **核心原则**：论证这个方向哪里可能错了
- **约束**：如果找不出有力质疑，必须承认"暂时无法有效反驳"
- **典型武器**：累库拐点、成本支撑、基差异常、产能过剩、需求崩塌

#### ⚪ 裁判/主持（闫判官）

- **职责**：辩论主持人+裁判。控时序、记论点树、5维评分、判胜负、尊重风控veto
- **4阶段流程**：
  1. **准备期**：spawn两份研究员→广播快照
  2. **辩论期**：6段时序控场（立论→rebuttal→自由交锋→final）
  3. **评审期**：收提案→判胜负→传策略师→处理veto
  4. **判决期**：5维评分→判决→归档
- **管控要点**：
  - 记录"对方提出但本方未回应"的待回应清单
  - 风控verdict=red时打回修改，不得判胜
  - 不允许以自己的观点影响评分

#### 📋 策略师（策执远）

- **职责**：把裁判判定的胜方提案翻译成可执行的交易方案
- **不改方向、不改目标价**，但可以调手数、改合约月份、加对冲腿
- **8工具清单**：
  1. `select_contract` — 合约选型（主力/次主/交割月检查）
  2. `size_position` — 凯利公式/固定分数模型算仓位
  3. `design_hedge` — 对冲方案（跨期/跨品种）
  4. `plan_entries` — 建仓方案（一次性/分批）
  5. `plan_stop` — 止损设计（ATR倍数/技术位）
  6. `plan_take` — 止盈目标
  7. `plan_roll` — 移仓计划（交割月前换月）
  8. `exit_triggers` — 动态退出条件
- **风控red回退**：被打回后修改方案，最多再审1轮

#### 🟡 风控（风控明）

- **角色定位三句话**：
  1. **站在"钱"那边** — 沙盘推演"按这个观点真开仓会怎样"
  2. **站在"逻辑"那边** — 戳穿"论证成立但仓位不成立"的坑
  3. **站在"规则"那边** — 定数据口径、合约月份、保证金假设
- **核心原则**：
  - 有veto（否决）权，但没有"改方向"权
  - 不能说"你空不对，应该多"；只能说"你这空法，仓位这么大要死"
- **8条红线**：
  - 杠杆 > 权益3倍 → **red**
  - 保证金占用 > 权益60% → **red**
  - 止损幅度 > 权益5%（单笔）→ **red**
  - 过夜持仓遇夜盘品种 → 必须跑simulate_gap
  - 交割月前5日未提移仓 → **yellow**
  - 尾部风险当基准情景 → **red**
  - 多空两腿品种相关系数>0.7却宣称"对冲" → **yellow**
  - 数据口径混用（主力连续/当月/指数）未声明 → **yellow**
- **6步决策链**：
  ```
  ①接结论 → ②口径校验 → ③算账(杠杆/止损/跳空) 
  → ④对冲自检 → ⑤逻辑质检 → ⑥出verdict(green/yellow/red)
  ```
- **7工具**：
  - `calc_position` — 按权益/保证金/止损反推开仓手数
  - `calc_leverage` — 算实际杠杆
  - `simulate_gap` — 历史夜盘跳空模拟
  - `check_correlation` — 多空品种相关性检查
  - `roll_check` — 交割月移仓检查
  - `stress_test` — 给定观点的P&L+追保路径
  - `flag_logic` — 叙事概率质检

## 三、五阶段工作流

### 3.1 时间线总览

| 时间 | 阶段 | 主导角色 | 关键产出 | 传给谁 |
|:----|:----:|:--------|:---------|:-------|
| T-60min | **选题与准备** | 🎯 明鉴秋 | 品种+周期+权益 | 全员广播 |
| T-50min | **数据准备** | 📡 数技源 | 原始数据包+方向 | 研究员 |
| T-40min | **研究出图** | 🟢 探源+观澜 | 两份快照(verdict:null) | ⚪ 闫判官 |
| T-30min | **广播** | ⚪ 闫判官 | 合并快照 | 全员 |
| T-30~T+0 | **辩论** | ⚪ 闫判官主持 | 正方v1→反方v1→正方v2 | ⚪ 闫判官 |
| T+0~T+5 | **判胜负** | ⚪ 闫判官 | 胜负+评分+待回应清单 | 📋 策执远 |
| T+5~T+15 | **策略合成** | 📋 策执远 | 可执行方案 | 🟡 风控明 |
| T+15~T+25 | **风控审核** | 🟡 风控明 | verdict+flags | ⚪ 闫判官+📋 策执远 |
| T+25~T+30 | **决策归档** | 🎯 明鉴秋 | 执行/搁置/重辩 | 交易台+知识库 |

### 3.2 阶段一：选题与准备（T-60min）

| 角色 | 动作 | 产出 | 传给 |
|:----|:----|:-----|:-----|
| 🎯 明鉴秋 | 选定品种+周期+账户权益假设 | `{subject, timeframe, equity}` | 全员广播 |
| 📡 数技源 | 调用 `scan_all.py --symbols` 采集K线+计算指标 | `数据包JSON`（含`_meta`/`data_manifest`） | 🟢 研究员 |
| 🟢 探源 | 基于原始数据+WebSearch，出基本面快照 | `fund_snapshot.json`（verdict:null） | ⚪ 闫判官 |
| 🟢 观澜 | 基于原始数据，出技术面快照 | `tech_snapshot.json`（verdict:null） | ⚪ 闫判官 |
| ⚪ 闫判官 | 合并两份快照，校验一致性 | `combined_snapshot.json` | 全员广播 |
| 🔵 证真/🔴 慎思 | 阅读快照，准备辩论 | — | — |

### 3.3 阶段二：辩论（T-30~T+0）

由闫判官主持6段时序：

| 时段 | 内容 | 裁判动作 | 产出 |
|:----|:-----|:---------|:-----|
| 0-8min | **正方立论**（证真） | 计时、记录论点树 | 正方论点清单 |
| 8-16min | **反方立论**（慎思） | 同上 | 反方论点清单 |
| 16-24min | **正方rebuttal** | 记录"哪些回了、哪些没回" | 待回应清单 |
| 24-32min | **反方rebuttal** | 同上 | 待回应清单 |
| 32-42min | **自由交锋** | 控每人≤2min，可call研究员补数据 | 新论点+证据更新 |
| 42-48min | **双方final**（各3min） | 计时，收最终提案 | 双方final提案 |

**最终提案格式**：
```json
{
  "side": "affirmative",  // 或 "opposition"
  "dimensions": [{"dim":"趋势结构","claim":"...","evidence":"...","confidence":0.8}],
  "scenarios": {
    "baseline": {"target": 2900, "probability": "60%", "desc": "..."},
    "upside": {"target": 3100, "probability": "20%", "desc": "..."}
  },
  "key_validation": "社库连续两周下降→认错",
  "confidence": 0.75
}
```

### 3.4 阶段三：策略合成（T+0~T+15）

```
⚪ 闫判官 → 5维评分 → 判胜负
    ↓ (传胜方提案给)
📋 策执远 → 合约选型 → 建仓方案 → 止损止盈 → 对冲检查 → 移仓计划
    ↓ (传可执行方案给)
🟡 风控明
```

策执远**不改方向不改目标价**，可调手数、改合约、加对冲腿。

### 3.5 阶段四：风控审核（T+15~T+25）

```
🟡 风控明 接收策执远方案
    ├─ 口径校验（合约月份、保证金）
    ├─ 算账（杠杆、止损幅度、跳空模拟）
    ├─ 对冲自检（品种相关性）
    ├─ 叙事质检（尾部当基准？数据打架？）
    └─ 出 verdict：
        ├─ GREEN → 放行
        ├─ YELLOW → 有条件放行（附要求）
        └─ RED → 打回策执远修改 → 再审（最多1轮）
    ↓ (传放行结果)
⚪ 闫判官 汇总
```

### 3.6 阶段五：决策归档（T+25~T+30）

| 角色 | 动作 | 产出 |
|:----|:-----|:-----|
| ⚪ 闫判官 | 出最终判决书（含评分+风控sign-off） | `final_judgment.json` |
| 🎯 明鉴秋 | 拍板 | **execute** / **hold** / **rematch** |
| 全员 | 所有论点+证据+方案+判决→知识库 | `debate_archive_{round_id}.json` |

## 五、决策机制与评分模型

### 5.1 闫判官5维评分模型

| 维度 | 权重 | 评分标准（1-10） |
|:-----|:----:|:----------------|
| **逻辑严谨度** | 30% | 论证有无断层、因果是否成立、有无偷换概念 |
| **证据充分性** | 25% | 是否引用研究员数据、数据口径是否正确、有无只靠叙事 |
| **反驳有效性** | 20% | 是否正面回应对方核心论点、有无遗漏重要反驳 |
| **风险意识** | 15% | 是否给出合理止损/仓位、是否区分基准与尾部情景 |
| **表达结构** | 10% | 论点是否清晰、层次是否分明、是否超时 |

> **总分 = Σ(维度分 × 权重)**，高分者胜。分差 < 5 分可判 draw。

### 5.2 裁决权重规则（最高原则）

**第0条：价格是唯一客观现实**（不可违反）

价格是各种要素（供需、政策、情绪、期限结构、资金流向等）作用的最终合力。所有其他分析维度都是对价格的解释，不是独立的投票成员。

- 价格行为是裁决的**唯一核心依据**
- 任何单一非价格维度独立说明力上限为**0**——不得作为方向裁决的独立依据
- 其他维度的正确用法：方向判断由价格信号做出后，用于**验证一致性和标注风险**

**第1条：右侧交易优先**

- ❌ 不得因任何非价格因素推翻价格走势的当前方向
- ✅ 非价格信号应标注为"值得关注的潜在风险/机会"，不改变裁决方向
- ✅ 裁决方向仅在出现右侧确认信号后才可转向

### 5.3 正反方辩论决策逻辑

**辩论焦点不是"多空谁更会讲故事"，而是"数技师的数据信号是否可靠"**：

```
数技师说 RB→做空 (ADX 59.5)
    ↓
正方（证真）：为什么空头方向正确？
    ├─ ADX 59.5 极强趋势不可逆
    ├─ 库存1623万吨+22万吨累库
    ├─ 产业链100%一致性
    └─ 螺纹产量由降转增+3.27万吨（反证供给收缩论）
    
反方（慎思）：哪里可能错了？
    ├─ RSI 27.9 超卖→空头衰竭？
    ├─ 成本3176元/吨，盘面3063跌破完全成本
    ├─ 电炉开工率持续下降
    └─ 基差+37现货升水不利于空头
    
正方v2（rebuttal）：逐条回应
    ├─ ADX>50时RSI可长期超卖钝化（承认超卖但否认衰竭）
    ├─ 产量数据反证供给收缩不成立
    └─ 现货端永钢下调100元/吨，基差收敛方向利好空头
```

### 5.4 风控verdict决策树

```
策执远方案传入
    │
    ├─ 杠杆 > 3倍？ → RED
    ├─ 保证金 > 60%权益？ → RED
    ├─ 止损 > 5%权益？ → RED
    ├─ 尾部事件当基准？ → RED
    │
    ├─ 交割月前5日未提移仓？ → YELLOW
    ├─ 数据口径混用未声明？ → YELLOW
    ├─ 对冲腿相关系数>0.7？ → YELLOW
    ├─ 含夜盘品种未跑simulate_gap？ → YELLOW
    │
    └─ 全部通过？ → GREEN
```

### 5.5 策执远仓位计算逻辑

使用**凯利公式**或**固定分数模型**：

```
凯利公式: f* = (bp - q) / b
  f* = 建议仓位比例
  b = 盈亏比 (预期收益/预期亏损)
  p = 胜率 (辩论置信度)
  q = 败率 (1 - p)

固定分数: 仓位 = 权益 × 固定比例 (通常1-2%)
```

**约束**：
- 总仓位不超过风控建议的30%
- 盈亏比≥1.5:1
- 止损使用ATR倍数（RB已验证1.8×ATR，其他默认1.5×ATR）

## 七、异常处理流程

### 7.1 风控连续2次Red

```
风控出RED → 策略师修改 → 再审 → 又出RED
    ↓
裁判暂停辩论
    ↓
团队主管（明鉴秋）召集三方会议
    ↓
明确分歧点 → 主管行使决策权
    ├─ 降级执行：降低仓位/改合约/加对冲 后通过
    ├─ 搁置：待关键数据（如面积报告）公布后再议
    └─ 重辩：裁判认为双方论证质量不够，从研究员阶段重新开始
```

### 7.2 辩手超时/离线

```
辩手超过30秒未响应
    ↓
裁判标记该阶段弃权 → 该阶段0分
    ↓
连续弃权2轮 → 考虑更换Agent
```

### 7.3 研究员数据延迟

```
研究员超过5分钟未出快照
    ↓
裁判5分钟催办
    ↓
超时未响应 → 跳过该研究员
    ↓
标注"该维度数据降级" → 置信度降一级
```

### 7.4 Agent输出不可达

```
Agent SendMessage 路由失败
    ↓
协调员读取优先级：
    ① 文件持久化产物 p{N}_{agent}_{trace_id}.json
    ② SendMessage 消息正文
    ③ 收件箱 inboxes/team-lead.json
    ↓
三项都读不到 → 直接调用 Python 库函数
    ├─ scan_all.py --symbols (数技师)
    └─ 链分析库函数 (产业链研究员)
```

## 八、数据溯源要求

### 8.1 核心原则

**每次辩论分析必须同步产出《数据资料溯源说明》**，作为报告组成部分，不得事后补充。

### 8.2 P1 K线数据溯源

每个品种的K线数据必须标注：
- **数据源**：如"通达信本地TQ-Local"
- **采集时间**：如"09:14"
- **最新K线日期**：如"2026-07-03(当天)"
- **数据条数**：如"170条"
- **技术指标计算方法**：如"基于以上K线数据，numpy向量化计算"

### 8.3 基本面数据溯源

每条证据的格式强制：
```
"{数值}（来源：XXX，截至X月X日，URL: XXX）"
```

禁止：
- ❌ 无来源的泛化表述如"库存偏高""需求疲弱"
- ❌ 无时间戳的陈述如"市场普遍认为"
