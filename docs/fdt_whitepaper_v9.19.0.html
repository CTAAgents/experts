FDT 期货辩论专家团

多Agent辩论的CTA决策系统 · 技术说明 v9.19.0

一、项目定位

FDT（Futures Debate Team）是一套面向期货中低频波段/隔夜策略、基于多Agent辩论实现实时推理的CTA决策系统。核心思路是通过多角色分工、交叉质询的方式，使决策过程可追溯、可审计。

系统在交易推理路径上与主流量化方法有所不同：

常见量化方法（归纳推理）： 基于历史数据统计规律拟合因子、回测择优，隐含假设是"历史模式在未来仍将重复"。这种方法在结构性突变行情（如2020年原油、2022年镍）中存在适应性不足的问题。

本系统思路（实时推演）： 不依赖历史行情复刻，而是聚焦当下——通过实时技术结构、基本面状态、产业链逻辑、新闻情绪，由多Agent辩论推演当前市场概率分布。决策依据来自实时信息链的逻辑自洽性，而非历史统计。这意味着每一次决策都是独立的，不因"过去类似时刻如何"而直接迁移。

需要说明的是，这种设计意味着系统不适用于传统量化回测框架——市场情绪、新闻语境、基本面场景往往具有即时性，难以通过历史数据还原。这是一个设计取舍，而非功能缺失。

2.3 数据引擎

系统通过 FDC（Futures Data Core） 整合多个数据源，按优先级自动降级。每个数据源配有独立熔断器（连续失败5次后暂停60秒）：

请求 → DataCore(0) — Data-Core统一接口 ↓ 失败 → TDX(1) — 通达信本地TQ-Local ↓ 失败 → Web(2) — 东方财富+新浪 ↓ 失败 → QMT(3) — QMT/xtquant ↓ 失败 → TqSDK(98) — 天勤量化(末位兜底) ↓ 全部失败 → 本地缓存(fdt_cache/) — SQLite增量缓存

数据覆盖：日线120天K线、实时报价、持仓排名、仓单、基差、跨期价差、期限结构、基本面(F10)、宏观数据、金十快讯/资讯/日历。

在P2.5阶段，系统会为入选辩论的品种预采集全量结构化数据，一次采集多次复用，减少后续阶段对数据源的重复请求。

三、设计特点

3.1 六阶段辩论流程

多空双方按固定流程完成攻防：

P4_1 多头立论(≥3条做多论据) P4_2 空头立论(≥3条做空论据) P4_3 空头反驳多头（交叉质询） P4_4 多头反驳空头（交叉质询） P4_5 空头结辩（最终陈词） P4_6 多头结辩（最终陈词） ↓ P5 闫判官六维评分裁决 + 风控审核 + 品藻质检

每条论据要求附带完整推理链（reasoning_chain/analysis_chain/risk_chain）和数据溯源信息（source/data_date/data_staleness_days）。数据超过5天不作为主论据使用。

3.2 量化策略的定位：算力减负工具

系统的核心是AI辩论推理。量化策略在此不是决策核心，而是一个可选的前置筛选工具。代码中预留了多策略框架，但生产环境目前只使用了通道突破（trend_following，含10个子信号）。

设计原则： 策略筛选的目的是过滤明显无效的震荡行情、减少辩论环节需要处理的品种数量，从而控制推理成本。筛选结果不干涉多空逻辑和最终裁决。

同时，系统提供了NO_FUSION机制：所有前置信号不融合、不加权，以原始状态进入辩论环节。这样设计是为了避免多信号对冲导致信息损失。此外还叠加了多层信号校验（震荡市识别ADX+BB+KF、去趋势Hurst+VR、P0-4伪突破拦截等共20+项检查）来减少误筛。

3.3 质量保障

  • P0-4 假突破拦截：六种验证器（raw_kline/volume_confirm/atr_vol_timing/trend_direction/stability/entity_quality）+ 全局 data_quality/crowding
  • 辩论输出质量治理（v9.14.0）：品藻实时质检（validate_argument/verdict/risk），不合格退回重修（最多2次）
  • 入口门禁：grade∈{STRONG,WATCH} 或 |total|≥40 进入辩论候选池
  • Maker-Checker 分离：裁决与审核由不同角色完成，降低自我验证偏差

3.4 多维数据融合

系统从四个维度获取信息：

  • 技术面（F1）：K线、均线、ADX、RSI、BB、成交量等，由观澜Agent处理
  • 基本面（F2）：库存、基差、利润、供需平衡表等，由探源Agent处理
  • 持仓资金（F3）：主力持仓、持仓量变化、净多空比
  • 宏观情绪（F4）：利率、贸易、地缘政策，叠加读心Agent新闻情绪分析

这种多维度设计是为了弥补单一量价数据在消息驱动型行情中的局限性。

3.5 运行模式

模式特点适用

4.1 Harness 规范体系

项目引入了 Harness Engineering 规范，覆盖六个维度：

配套12项 commit 前检查、10条反模式检测规则。

4.2 Loop Engineering：三层循环体系

FDT 的自动执行链路实际上由三个嵌套的"循环"构成，每个循环有明确的触发条件、执行范围、预算约束、停止条件和输出报告——即 Loop Contract（循环契约）。

第一层：品种级重修循环（最内层）

在单次辩论会话中，每个品种完成 P3→P4→P5 后进入品藻质检。如果输出不合格且重试次数未达上限（2次），系统将该品种退回 P2.5 重新处理，而非直接丢弃。这个微型循环确保辩论输出满足基本质量标准后才进入下一步。

第二层：辩论执行循环（Inner Loop）

完整的 P1→P2→逐品种循环→P6 流程构成一次辩论会话。Master Graph 在设定的时间窗口（如工作日19:15）触发 daily-debate 任务，所有品种处理完成后输出报告和 CTP 信号，会话终止。该循环的停止条件是"所有选中品种全部完成处理"。

第三层：自进化循环（Outer Loop）

辩论会话结束后，Evolution Graph 自动启动：收集质量指标 → APM五轴评分 → 按条件触发后续步骤（APM退化则自改进、样本充足则校准权重、再充足则进化 Agent、积累足够则增量训练 ML）。该循环的停止条件是"所有条件均已评估完毕，无更多操作可执行"。

三个循环通过 Loop Contract 进行契约化管理，每份契约定义六项要素：

4.2 自进化闭环

系统包含一个自动化的质量反馈循环：

┌─ 实时层（每轮辩论） │ 品藻质检 (validate_argument/verdict/risk) │ D3结构化输出校验 + D6质量评分 ├─ 反馈层（按条件触发） │ T+1验证 → 权重校准(≥5条有效) │ → Agent进化(≥5条样本) → ML训练(≥50条样本) ├─ 监控层（周期性） │ 每周APM五轴评分(D1-D5) │ 失败聚类分析 ├─ 改进层（持续） │ 自改进提案 → 解码配置微调 │ 输出审计 → 报告模板优化 │ 异常检测 → 工具调用优化

这个循环不依赖人工标注，以实际行情反馈作为评估依据。

4.4 逐Agent LLM 配置

系统允许按角色独立配置LLM参数（API Key/BaseURL/Model），配合 D3 解码控制，配置优先级为：decode_config.yaml > 逐Agent环境变量 > 全局环境变量 > 默认值。

六、待改进方向

以下问题属于工程层面的完善空间，不影响系统顶层设计思路。

6.1 数据源容灾

系统已内置五级数据降级链（DataCore→TDX→Web→QMT→TqSDK），核心行情数据具备多层容错。但金十MCP作为新闻情绪和快讯的独立数据源，尚无同等完善的降级兜底方案。金十接口异常时，读心Agent的情绪分析维度会受影响，其余数据链路不受干扰。

6.2 风控精细化

现有风控框架（green/yellow/red三级）偏基础，缺少动态仓位管理、阶梯止盈止损、最大回撤限制、单日交易限额、极端行情熔断等实盘所需规则，不适合全自动运行。

6.3 落地工具链

技术文档相对完整，但入门部署、环境配置、参数调优、问题排查方面的实操指南较少，上手门槛偏高。

七、结语

FDT 是一个以多Agent辩论为核心机制的CTA决策系统。相较于传统量化依赖历史统计归纳的方法，它选择了一条不同的技术路径——通过实时信息的多角色交叉验证来形成判断。

系统已具备基本的工程完备性（模块化设计、自动化调度、质量门禁、自进化机制），单次全量推理成本在可控范围内。现存的问题主要集中在工程细节和运维工具链上，不影响核心决策逻辑。

这个项目更多地代表了作者对"AI+量化"这个方向的一种技术探索——不一定是答案，但希望提供一种有意义的参考。
