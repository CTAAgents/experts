# Loop Contracts — FDT 循环契约规范

> FDT 的每一个自动化循环（Loop）都必须有一份明确的契约（Contract），定义其触发条件、作用范围、具体行为、预算红线、停止条件和上报通道。契约是机器可读的，也是人可读的。

---

## 为什么需要 Loop Contract

FDT 从 v8.0 开始去 WorkBuddy 依赖，演进为独立运行的多 Agent 系统。随着自动化循环增多（每日辩论、自进化、数据采集、ML 训练等），我们面临三个核心问题：

1. **隐性配置**：循环的触发、范围、预算散落在代码、配置文件、记忆文件中，新人无法快速理解
2. **边界模糊**：每个循环能碰什么、不能碰什么，没有显式声明
3. **失控风险**：没有统一的预算红线和停止条件，循环可能空转或越界

**Loop Contract 就是把这些隐性知识显性化，用统一格式说清楚每个循环的"是什么、做什么、不能做什么、花多少钱、什么时候停、出问题找谁"。**

---

## 六维度契约格式

每个 Loop Contract 包含六个维度，遵循 TRIGGER / SCOPE / ACTION / BUDGET / STOP / REPORT 框架：

| 维度 | 键 | 说明 | 必填 |
|------|-----|------|:----:|
| **触发条件** | `trigger` | 何时启动、由谁启动 | ✅ |
| **作用范围** | `scope` | 能碰什么（allow）、不能碰什么（deny） | ✅ |
| **具体行为** | `action` | 调醒之后干什么，pipeline 步骤 | ✅ |
| **预算红线** | `budget` | Token、时间、成本、重试次数上限 | ✅ |
| **停止条件** | `stop` | 正常完成条件 + 安全网 | ✅ |
| **上报通道** | `report` | 正常输出 + 异常通知 + 升级路径 | ✅ |

## 验证档位

每个循环必须声明当前验证档位（与 FDT L1-L5 鲁棒性层对应）：

| 档位 | 验证方式 | 适用场景 |
|------|----------|----------|
| **L1 (self)** | Agent 自检、JSON Schema 校验 | 报告型循环、低风险 |
| **L2 (test_suite)** | pytest 测试套件、脚本化验证 | 中等风险、有测试保障 |
| **L3 (independent_agent)** | 独立 Agent 审查（闫判官+风控明模式） | 高风险、不可自我验证 |

### 晋级门槛

| 晋级 | 要求 |
|------|------|
| L1 → L2 | 影子模式 ≥5 轮，准确率 ≥90%；金标准任务全通过 |
| L2 → L3 | 连续 ≥20 次零回退；验证器漏放率 ≈0；人工干预率 <10% |
| 任一级 → 降级 | 出现一次安全事件，或人工干预率连续两周 >30% |

## 权限三档

与验证档位匹配的权限等级：

| 档位 | 权限 | 对应验证档位 |
|------|------|:------------:|
| **只读（RO）** | 读文件、跑只读命令、写报告 | L1 |
| **草稿（Draft）** | 上述 + 写隔离区（workspace、state、开 Draft PR） | L2 |
| **外部写入（Write）** | 上述 + CTP 信号输出、外部 API 调用、文件修改 | L3（需逐项准入） |

- [ ] ① `trigger` 至少一种触发方式已配置
- [ ] ② `scope.allow` + `scope.deny` 已明确定义
- [ ] ③ `action.pipeline` 步骤清单 + 每步超时已定义
- [ ] ④ `budget` 四维（token/时间/成本/重试）已设置硬上限
- [ ] ⑤ `stop.completion` + `stop.safety` 停止条件齐全
- [ ] ⑥ `verification.mode` 已声明（L1/L2/L3）
- [ ] ⑦ `report.success` + `report.failure` + `report.escalation` 齐全
- [ ] ⑧ 已登记到本 README 的循环清单表
- [ ] ⑨ 已通过四步准入评估（影子模式 → 金标准 → 验证器度量 → 金丝雀）

## 数据新鲜度原则（通用 — 所有循环必须遵守）

数据新鲜度是所有分析工作的基础。过时的数据无论分析多深入，结论都可能被市场证伪。每个循环必须遵守以下新鲜度标准：

### 新鲜度分级标准

| 数据类型 | 最大可接受延迟 | 优先级 | 说明 |
|----------|:-------------:|:------:|------|
| **行情价格（日盘/夜盘）** | **上一交易日** | P0 | 超过2个交易日不可用于现价可操作性判断 |
| **资金流向/持仓** | **最新交易日** | P0 | 必须使用盘后快讯或日终数据 |
| **技术指标（支撑/阻力）** | **最新收盘价** | P0 | 基于当前价格计算，不使用过时点位 |
| **新闻/情绪** | **当日/前日** | P1 | 搜索时使用日期锚定词（如当天日期） |
| **基本面（库存/开工率）** | **周度最新** | P1 | 优先使用周报/日报，月报仅做背景框架 |
| **机构观点/研报** | **1周内** | P2 | 过时观点不作为核心论据 |

### 新鲜度检查清单（每条数据引用前执行）

- [ ] 该数据的采集时间是否在最大可接受延迟范围内？
- [ ] 如果超过阈值，是否有更新的同源数据可替代？
- [ ] 如果无可替代，是否在报告中标注了"数据采集时间"？
- [ ] 核心结论是否依赖过时数据？如果是，重新评估结论有效性

### 新鲜度不达标处理规则

1. **行情/资金数据不达标**：禁止用于现价可操作性判断，宁可不引用也不拿旧数据充论据
2. **基本面数据不达标**：仅作为背景参考，明确标注数据日期，不用于短期交易决策
3. **所有循环执行前**：必须执行 data-collection 循环的 `check_freshness` 步骤，确保基础数据新鲜
4. **辩论/裁决环节**：任何分析 Agent 如发现引用数据已过时，有权拒绝执行并要求重新采集

### 可观测性指标

每个循环必须上报以下新鲜度指标：
- `data_freshness_hours`：各品种最新数据距离当前的小时数
- `stale_data_ratio`：本次循环中过时数据引用占比
- `freshness_block_count`：因新鲜度不达标被拦截的执行步骤数
