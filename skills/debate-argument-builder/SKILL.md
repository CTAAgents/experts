# 通用期货论点构建器

## 依赖
- **输入方**：`BullOutput` / `BearOutput`（`contracts/debate.py`）
- **上游依赖**：依赖 P1 `DataCollectionOutput`（data）、P1 `TechnicalOutput`（tech）、P2 `ChainAnalysisOutput`（chain）
- **版本**：`2.0`（所有 schema 带 `version` 字段，编排层 `parse_and_migrate` 自动路由）
- **输出方式**：正文（Markdown 人类可读）+ 末尾 ```json fence 结构化摘要

## 概述

LLM推理 + 主动基本面搜索。基于结构化市场数据构建多空定性分析论点，同时**主动使用 WebSearch/WebFetch 搜索基本面信息**补充论证深度。

**可复用于**：期货交易辩论专家团、自定义分析管道、研究报告生成、交易日志分析等。

## 角色模式

| 模式 | 方向 | 输出 |
|:----:|:----:|------|
| **bull** | 🟢 多头 | 做多看涨论点 |
| **bear** | 🔴 空头 | 做空看跌论点 |

## 独立使用模式

### 输入格式

接受任意品种的结构化市场数据：

```json
{
  "symbols": [
    {
      "id": "品种代码",
      "direction": "BUY/SELL",
      "price": 当前价格,
      "technical": {
        "trend_stage": "启动/主升/主跌/衰竭/反转",
        "ma_alignment": "多头/空头/交叉/震荡",
        "adx": 数值,
        "rsi": 数值
      },
      "volume_price": {
        "oi_trend": "增/减/稳",
        "price_volume": "价涨量增/价涨量缩/价跌量增/价跌量缩",
        "basis": "走强/走弱/平稳"
      },
      "term_structure": "contango/back/flat",
      "chain": {
        "name": "产业链名",
        "consistency": 0-100
      },
      "data_quality": "正常/降级/缺失"
    }
  ]
}
```

### 输出格式

采用**双轨输出**格式：

**① 正文（人类可读）**：用连贯 Markdown 正文撰写，6 维度分析框架，附具体数字和引用来源。

**② 结构化摘要（```json fence 机器消费）**：在正文末尾另起一行，用 ```json fence 输出以下字段，字段名不可改：

```json
{
  "variant": "bull",                          // bull 或 bear，固定值
  "dimensions": [
    {"dim": "供给", "claim": "核心观点", "evidence": "仓单环比-12%", "confidence": 0.8},
    {"dim": "需求", "claim": "核心观点", "evidence": "同比+5%", "confidence": 0.6}
  ],
  "summary_4_risk": "供给收缩+基差走强，主要风险在...",  // ≤100字，给风控的精简版
  "full_text": "见上方正文",                              // 固定值，与正文一致
  "confidence": 0.76,                                     // 整体置信度0-1
  "rebuttal_targets": []                                   // 本轮反驳了熊的哪些维度，空数组表示首轮
}
```

**解析方式**：下游从 ```json fence 中 parse 结构化数据，正文保留给 HTML 报告/人类阅读。

### 独立调用示例

```bash

在正文末尾另起一行，用 ```json fence 输出以下字段，字段名不可改：

```json
{
  "variant": "bull",                          // bull 或 bear，固定值
  "dimensions": [
    {"dim": "供给", "claim": "核心观点", "evidence": "仓单环比-12%（来源：上期所，截至6月30日）", "confidence": 0.8}

  "confidence": 0.76,                                     // 整体置信度0-1
  "rebuttal_targets": [],                                  // 本轮反驳了对手的哪些维度，空数组表示首轮
  "scenarios": {                                           // 场景分离（v2新增）

    "bull_case": {"target": 4200, "probability": "20%", "desc": "乐观情景：限产超预期"},
    "bear_case": {"target": 3400, "probability": "20%", "desc": "悲观情景：需求崩塌"}

### Pydantic Schema 定义

当本 skill 被辩论专家团集成使用时，按以下 schema 验证：

```python
from pydantic import BaseModel
from typing import Literal

class DimensionItem(BaseModel):
    """单维度分析项"""
    dim: str                      # 维度名称
    claim: str                    # 核心观点
    evidence: str                 # 可核验证据（必含具体数字，格式：`"数值（来源：XXX，截至X月X日）"`）
    confidence: float             # 该维度置信度 0-1

class AffirmativeSchema(BaseModel):
    """证真（正方辩手）论点。variant="affirmative"：论证数技师方向的正确性"""
    variant: Literal["affirmative", "opposition"]
    dimensions: list[DimensionItem]
    summary_4_risk: str           # ≤100字，给风控的精简摘要
    full_text: str                # "见上方正文"
    confidence: float             # 整体置信度 0-1
    rebuttal_targets: list[str]   # 本轮反驳了对手的哪些维度名，首轮为[]
```

**产出规范（双轨输出）**：
- Agent 产出同一份消息包含：① Markdown 正文（人类可读，给 HTML 报告） ② 末尾 ```json fence 结构化摘要（给明鉴秋机器消费）
- 明鉴秋从 ```json fence 中 parse 结构化数据，正文保留给 HTML 报告
- 下游通过 `obj.dimensions`、`obj.summary_4_risk` 等属性访问
- `full_text` 固定为"见上方正文"（结构化摘要不重复正文，正文在 fence 外保留）
- 过渡期兼容：如果明鉴秋扒不到 JSON fence，走旧路径将全文当文本传递

## 辩论专家团集成模式

当被 `futures-trading-analysis` 辩论系统的 **正方辩手（证真）** 或 **反方辩手（慎思）** Agent 加载时：

**输入**：由 明鉴秋 传入辩论候选品种的 typed 结构化数据（`DebateState` 的 `data`/`tech`/`chain` 字段）
**产出**：正文（人类可读）+ 末尾 ```json fence 结构化摘要 → SendMessage → main
（明鉴秋从 fence 扒结构化数据，正文保留给 HTML 报告。过渡期保持双轨，旧路径继续可用。）

**Prompt 硬约束（L2 辩论铁律）**：
当本 skill 被辩论专家团集成使用时，以下五条硬约束必须嵌入 system prompt：

1. **禁止附和语**：回应对手时 ❌ 禁止用"看多/空方说得有道理""你补充得很好""但是反过来"开头。必须直入主题"熊曰[X维度：证据] → 牛驳：反证"
2. **证据强制**：每个维度 ≥1 个可核验数字（合约价、基差、仓单、持仓）。没有可引用的具体数字时，该维度标注"数据不足"并降置信度
   - **证据格式强制**：每个 evidence 字段必须包含来源+日期，格式如下：
     ```
     "evidence": "具体数值+单位（来源：XXX机构/媒体，截至X月X日）"
     ```
     示例：`"五大钢材总库存1601.99万吨（来源：Mysteel周度数据，截至6月30日）"`
   - 违反此格式的 evidence 算作不合格，风控明必须将其标注为"糊弄"
3. **角色锚定**：正方辩手（证真）为"信号捍卫者——数技师说多就论证多，说空就论证空"，反方辩手（慎思）为"信号挑战者——站在数技师方向的对立面找漏洞"。双方立场由数据决定，不是预设的多/空。
4. **场景分离**：必须将"基准情景（baseline）"和"乐观/悲观情景（bull/bear case）"分开标注，不得混为一谈
   - `scenarios.baseline`: 最可能的情景（概率>50%）
   - `scenarios.bull_case` / `scenarios.bear_case`: 有利/不利的尾部情景
5. **认错信号（key_validation）**：每个 final proposal 必须附带一个"什么条件下我认错"的触发条件，如"厂库环比连升3周且back翻contango → 多单离场"

**重要 — 全品种覆盖规则**：
- 正方辩手必须对**每个品种**都站在数技师指示的方向上论证
- 反方辩手必须对**每个品种**都站在数技师指示的对立面上质疑
- 正方辩手论证"这个方向为什么对"
- 反方辩手论证"这个方向哪里可能错"

**交叉质询模式**：
- 步1：正方辩手首轮产出 `affirmative_v1`（立论：方向是正确的）
- 步2：反方辩手读 `affirmative_v1` 后产 `opposition_v1`（质疑：方向哪里不对）
- 步3：正方辩手读 `opposition_v1` 后产 `affirmative_v2`（rebuttal，max=1）
- 终止条件：如果正方v2对反方v1的rebuttal里 ≥3/5 维度承认"这点质疑成立，但…" → 可提前结束
- 非对称性：反方只产一次（opposition_v1），第二回合是正方rebuttal（affirmative_v2）

**基本面搜索**：必须在分析过程中主动使用 WebSearch/WebFetch 搜索以下基本面信息：
- 品种最新新闻（政策变化、行业事件）
- 供需数据（库存报告、产量数据、进出口）
- 市场情绪（持仓分析、资金流向、市场观点）
- 基差/现货走势
- 产业政策动向

## 工作流程

1. 读取 data + tech + chain 的数据
2. 逐维度分析（供给、需求、库存、基差、宏观、产业链、市场情绪等）
3. **如果是第二轮（有 rebuttal_targets 输入）**：先写 Rebuttal 段，再写己方更新版。禁止用"你说得对""但是反过来"开头，必须有数据支撑
4. 输出完整正文 + 末尾 ```json fence 结构化摘要

| 维度 | 多头（做多） | 空头（做空） |
|------|-------------|-------------|
| **1. 趋势结构** | 通道突破上轨、MA多头排列、MACD金叉、Higher High | 通道跌破下轨、MA空头排列、MACD死叉、Lower Low |
| **2. 量价关系** | OI增+价涨→健康；基差走强→现货支撑 | OI减+价跌→溃散；基差走弱→现货疲软 |
| **3. 期限结构** | Back→做多有利，近月溢价 | Contango→做空有利，展仓收益 |
| **4. 产业链验证** | 方向一致→加强；背离→需合理解释 | 同左（对称） |
| **5. 基本面/市场情绪** | 通过WebSearch/WebFetch搜索当前新闻、供需报告、政策动向、库存数据，验证基本面是否支持方向 | 同左（对称），搜索空头基本面因素 |
| **6. 风险点** | 至少列出2个风险 | 至少列出2个风险 |

## ⚠️ 数据来源铁律（全局强制）

**所有辩论证据（基本面数据、新闻、库存、供需、政策等）必须满足以下条件，否则不得使用：**

1. **真实来源**：必须来自可追溯的公开信息源——权威新闻媒体（新华财经、路透、Bloomberg等）、官方机构公告（交易所、统计局、行业协会官网）、经核实的研究报告。**不得使用猜测性文章、匿名爆料、自媒体未核实信息。**
2. **引用可查证**：每个论据必须附带来源引用（如"数据来自X月X日XX机构半年报"或"根据6月X日XX新闻"），**不得出现无来源的泛化陈述**（如"众所周知""市场普遍认为"）。
   - **evidence字段格式校验规则**（2026-07-03新增）：evidence字符串必须匹配正则 `/（来源：.+[，，]截至.+[月日]）/` 或 `/（来源：.+[。]）/`，否则该维度由风控自动标记为"糊弄"。
3. **多方验证优先**：对关键数据（库存、产量、政策变动），优先使用多个独立来源交叉验证的数据。单一来源的数据需标注"单一来源"并降低置信度。
4. **禁止编造**：**严禁编造模拟数据、虚构新闻报道、伪造官方公告、生成虚假库存/产量数字。** WebSearch搜索不到有效数据时，必须如实报告"当前搜索未找到近期数据"，不得用LLM内部知识替代。
5. **置信度挂钩**：数据来源质量直接影响论点置信度——多方验证=高置信度；单一权威来源=中置信度；无法搜索到数据=低置信度并标注"缺乏近期基本面数据支撑"。

## ⏱️ 资料时效性铁律（全局强制）

**所有 WebSearch/WebFetch 获取的基本面信息，必须按时效性分级处理：**

### 时效窗口定义

| 数据类别 | 有效窗口 | 超窗处理 | 示例 |
|---------|:------:|---------|------|
| **价格/行情/盘面数据** | ≤1天 | 超窗直接丢弃 | 收盘价、涨跌幅、基差 |
| **突发新闻/事件** | ≤3天 | 标注"X天前"+降置信度 | 政策发布会、矿山事故、天气预警 |
| **周度数据**（库存/开工率/持仓） | ≤7天 | 标注发布日期间隔 | EIA库存、Mysteel周报、CFTC持仓 |
| **月度数据**（产量/进出口/PMI） | ≤31天 | 标注月份+降一级 | 海关进出口、统计局产量、PMI |
| **季度/半年报**（机构展望/行业报告） | ≤45天 | 标注"基于X月X日报告，可能已过时" | 年报、半年展望、政策白皮书 |
| **政策/法规**（产能红线/关税/配额） | ≤90天 | 标注发布时间，核查是否有更新 | 印尼RKAB配额、出口退税、产能红线 |

### 时序搜索规范

1. **搜索查询必须包含时间限定**：`"{品种名} {关键词} 2026年6月 最新"` 或 `"{品种名} {关键词} 2026-06-25..2026-07-01"`
2. **取最新可用数据**：同一指标有多个时间版本时，取日期最新的数据，并在引用中标注数据截止日期
3. **超窗数据仍可用但必须降级**：
   - 7天内数据 → 正常使用
   - 8-14天数据 → 使用但标注"截至X月X日，已间隔Y天"
   - 15-30天数据 → 降置信度一级 + 标注"数据时效性较差"
   - >30天数据 → 搜索更新版本；搜不到则标注"仅找到X月X日历史数据，可能已过时"并使用LLM内部知识判断方向趋势，不得使用旧数据做精确数值论证
4. **数据截止日期必须写入 evidence**：如 "纯碱库存178.36万吨（截至6月30日，周度数据）" 而非 "纯碱库存178.36万吨"
5. **对比验证**：对时效性敏感的关键数据（库存、基差、持仓），搜索确认是否为"最新一周/月"数据——搜索"纯碱 库存 2026年7月1日"确认没有更新的周报已发布

### WebSearch 获取数据后的处理

```
搜索到数据 → 检查发布日期 → 
  ├─ 时效窗口内 → 直接使用，标注日期
  ├─ 轻微超窗 → 使用但标注"已间隔Y天"
  ├─ 严重超窗 → 降置信度+尝试重新搜索更新版本
  └─ 搜索不到 → 标注"缺乏近期×(库存/新闻/政策)数据支撑"
```

## 基本面搜索指引

对每个品种，至少执行 2-3 次 WebSearch 查询，**每次查询必须包含时间限定词**：

```
# 例如对 rb（螺纹钢）：
查询1: "螺纹钢 库存 2026年7月 最新 周度" → 了解最新库存水平
查询2: "螺纹钢 需求 基建 房地产 2026年6月" → 了解需求侧近况
查询3: "螺纹钢 政策 限产 环保 2026" → 了解供给侧政策
```

### 降级条件

- 数据质量降级 → 置信度降一级
- ADX < 15 → 标注"震荡偏多/空"
- RSI > 80(做多)或 < 20(做空) → 标注"追高风险"
- 同链高相关冗余 → 标注"与XXX重复"
- **WebSearch 搜索不到有效的近期基本面数据** → 置信度降一级，标注"缺乏近期基本面支撑"
